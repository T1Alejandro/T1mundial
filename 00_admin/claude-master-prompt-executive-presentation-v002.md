# Prompt maestro para Claude

Copia y pega este prompt en Claude Code desde la raíz del repo:

```text
Actúa como productor ejecutivo AI-first, director estratégico de producción cinematográfica, director editorial tipo Apple/Figma keynote y diseñador senior de presentaciones ejecutivas.

Necesito crear una presentación ejecutiva / PRD visual en PDF para explicar a dirección la siguiente decisión de producción del proyecto “T1 Mundial”.

El objetivo NO es hacer un documento técnico largo.
El objetivo es crear una presentación premium, ultra clara y elegante, que permita entender en minutos por qué necesitamos cubrir correctamente la fase final de producción AI-video.

## Proyecto

Nombre: T1 Mundial
Repo local: `/Users/parco/T1mundial`

Trabaja desde esta carpeta.

## Qué necesito comunicar

En máximo una idea:

T1 Mundial ya no está en exploración conceptual. Ya existe base visual, narrativa, stills, model sheets, pipeline y estructura operativa. Ahora la decisión crítica es cubrir correctamente la ventana de producción para terminar 34-36 tomas cinematográficas sin perder continuidad, control ni flujo creativo.

El mensaje NO debe sonar a:

- “necesito más créditos”,
- “quiero probar otra herramienta”,
- “no sé qué usar”,
- “estamos improvisando”.

Debe sonar a:

- “El proyecto ya está maduro”.
- “La producción AI-video necesita margen operativo real”.
- “El costo real vive en iteraciones, no en segundos finales”.
- “Cada herramienta debe tener un rol preciso”.
- “La decisión correcta reduce riesgo, desgaste y retrabajo”.

## Contexto real

El proyecto consiste en aproximadamente 34 a 36 tomas cinematográficas de 3 a 5 segundos cada una, para construir un video final cercano a 1 minuto.

Ya existen:

- stills base,
- rack visual vigente,
- estructura narrativa,
- folders organizados,
- pipeline operativo,
- prompts,
- assets,
- model sheets,
- referencias,
- estructura técnica trabajada desde Claude Code + VSCode.

La preocupación principal no es solo el costo económico.

También importan:

- estabilidad operativa,
- claridad del pipeline,
- continuidad visual,
- velocidad de iteración,
- desgaste mental,
- control legal,
- evitar quedarnos sin créditos a mitad de producción,
- evitar elegir herramienta por hype.

## Lo que ya entendimos

La decisión ya NO es:

“qué herramienta está más cool”.

La decisión real es:

qué pipeline permite terminar el video completo con:

- menos fricción,
- menos desgaste mental,
- suficiente libertad creativa,
- continuidad visual,
- estabilidad operativa,
- costo relativamente predecible,
- margen suficiente para iterar.

## Herramientas consideradas

### Claude Code + VSCode

Rol:
Sistema operativo del proyecto.

Sirve para:

- estructura,
- documentación,
- prompts,
- PRD,
- shot tracking,
- naming,
- templates,
- control de producción,
- QA textual,
- coordinación del sistema.

### Krea AI

Percepción:
Se siente como creative operating system.

Pros:

- buena experiencia operativa,
- node editor,
- exploración visual rápida,
- flujo creativo claro,
- styles,
- organización visual cómoda,
- menor fricción mental.

Contras:

- no siempre llega al nivel cinematográfico de Higgsfield.

Uso ideal:
Hub creativo, exploración, node workflows, organización visual, look development.

### Higgsfield

Percepción:
Muy fuerte cinematográficamente.

Pros:

- excelente lenguaje cinematográfico,
- cámara y movimiento,
- tomas complejas,
- shots hero,
- sensación premium.

Contras:

- flujo disperso,
- más carga mental,
- puede fragmentar la operación,
- ya se agotaron créditos Pro.

Uso ideal:
Motor cinematográfico principal para tomas hero y motion sofisticado.

También considerar que a futuro podría ser útil para el equipo completo si necesitan crear contenido cinematográfico.

### Seedance

Percepción:
Debe evaluarse como motor fuerte de generación/render, no como espacio principal para pensar.

Uso esperado:

- generación final,
- stitching,
- render,
- producción de clips,
- capa final de producción.

Pregunta central:
Cómo calcular cobertura suficiente para 34-36 tomas sin tener que volver a comprar créditos a mitad de producción.

### Figma Weave

Importante:
Validar con información actual si Figma Weave viene de Weavy y fue adquirido/integrado por Figma. Si puedes verificarlo con internet, hazlo. Si no puedes verificarlo, trátalo como hipótesis operativa.

Percepción:

- UX muy cuidada,
- filosofía cercana a Figma,
- lógica de nodos,
- workflows creativos visuales,
- se siente humano y pensado,
- puede explotar entre diseñadores,
- muy prometedor para exploración y R&D.

Pero:
No lo pongas todavía como columna vertebral absoluta de la producción final hasta validarlo con pruebas reales.

Uso ideal por ahora:
Exploración visual avanzada, workflows personales, R&D creativo, pruebas de flujo, posible capa futura de orquestación.

Idea importante:
Weave podría pagarlo/operarlo Arturo personalmente como capa de exploración avanzada, mientras Higgsfield/Krea/Seedance se plantean como herramientas más compartibles o productivas para el equipo.

## Reflexión clave de costo

El error sería calcular:

34 tomas x 4 segundos.

Eso no refleja producción AI-video.

El cálculo real es:

tomas x iteraciones.

Rangos base:

- 34 tomas x 4 iteraciones = 136 generaciones
- 34 tomas x 6 iteraciones = 204 generaciones
- 34 tomas x 8 iteraciones = 272 generaciones

Si se usan 36 tomas:

- 36 x 4 = 144 generaciones
- 36 x 6 = 216 generaciones
- 36 x 8 = 288 generaciones

El costo real viene de:

- refinamiento,
- continuidad,
- motion,
- fixing,
- consistencia,
- stitching cinematográfico,
- descartes,
- regeneraciones,
- legal cleanup,
- pruebas de herramienta.

No inventes precios actuales si no puedes verificarlos.
Si propones costos, hazlo como modelo de escenarios y supuestos.

## Tiers de producción

Incluye una matriz simple:

### Lean

- 4 iteraciones promedio por toma.
- 136 generaciones para 34 tomas.
- 144 generaciones para 36 tomas.
- Menor costo, mayor riesgo de continuidad.

### Controlled

- 6 iteraciones promedio por toma.
- 204 generaciones para 34 tomas.
- 216 generaciones para 36 tomas.
- Recomendado para producción seria.

### Full Coverage

- 8 iteraciones promedio por toma.
- 272 generaciones para 34 tomas.
- 288 generaciones para 36 tomas.
- Más margen, menor fatiga, mayor protección de calidad.

No pongas precio total si no tienes precio verificado por generación/crédito.
Si hay que hablar de costo, usa “cobertura” y “ventana operativa”.

## Decisión que debe facilitar

La presentación debe ayudar a dirección a decidir entre:

1. Aprobar un sprint controlado de validación por herramienta.
2. Aprobar cobertura suficiente para producción completa.
3. Decidir si el stack final será:
   - Higgsfield-led,
   - Seedance-led,
   - Krea/Weave orchestration-led,
   - o híbrido por tipo de toma.

Mi recomendación preferida:

Híbrido controlado:

- Krea/Weave para organización y exploración.
- Higgsfield para tomas hero/cinematográficas.
- Seedance para generación/render y cobertura.
- Claude Code + VSCode como control operativo.
- Post/Adobe para pantallas, cleanup, color, cierre y master.

## Qué no quiero

No quiero una presentación que parezca:

- “estoy confundido y necesito que mi jefe decida todo”,
- “me acabé créditos y necesito más”,
- “compramos otra app porque está cool”,
- “documento de ingeniería”.

Quiero que parezca:

- “ya hicimos el trabajo serio de estructura”,
- “sabemos dónde está el riesgo”,
- “tenemos opciones claras”,
- “la recomendación protege calidad, recursos y foco”.

## Definición de éxito

El resultado debe permitir presentar esto en 5-8 minutos y que dirección entienda:

- qué ya está avanzado,
- por qué no basta calcular 34 x 4 segundos,
- por qué la iteración es el costo real,
- por qué una sola herramienta no necesariamente resuelve todo,
- por qué el pipeline híbrido reduce riesgo,
- qué decisión concreta se necesita ahora.
```
